# Photo Storage

FoundFlow stores photo bytes outside service databases. Services persist only
the generated `photoKey`.

## Consumers

| Service | Photo rule |
| --- | --- |
| `lost-item-service` | Optional single photo per lost report |
| `found-item-service` | Required photo per found item |

## Shared Module

The contract and adapters live in `shared/photo-storage`.

Main types:

- `PhotoStorage`
- `PhotoData`
- `PhotoConstraints`
- `PhotoUrlResponse`
- `PhotoStorageException`
- `PhotoNotFoundException`

The business services inject `PhotoStorage`; they do not branch on MinIO,
Azure, or local filesystem providers.

## Providers

| Provider | Use |
| --- | --- |
| `minio` | Local Docker Compose and local Kubernetes |
| `azure` | Cloud Blob Storage-compatible deployment |
| `local` | Lightweight tests and local fallback |

Configuration:

```properties
PHOTO_STORAGE_PROVIDER=minio
PHOTO_STORAGE_SIGNED_URL_TTL=PT10M
PHOTO_STORAGE_ENDPOINT=http://minio:9000
PHOTO_STORAGE_PUBLIC_ENDPOINT=http://localhost:9000
PHOTO_STORAGE_ACCESS_KEY=...
PHOTO_STORAGE_SECRET_KEY=...
PHOTO_STORAGE_BUCKET=foundflow-found-photos
```

Lost and found services use separate buckets/containers:

- `foundflow-lost-photos`
- `foundflow-found-photos`

## Object Keys

Keys are generated by the storage module, not by services.

```text
{domain}/{yyyy}/{MM}/{uuid4}.{ext}
```

Examples:

- `lost-reports/2026/06/00000000-0000-4000-8000-000000000000.jpg`
- `found-items/2026/06/00000000-0000-4000-8000-000000000000.webp`

Services only serve photos via keys already stored on authorized records.
Callers cannot provide arbitrary keys to retrieve objects.

## Constraints

| Constraint | Value |
| --- | --- |
| Allowed content types | `image/jpeg`, `image/png`, `image/webp` |
| Maximum size | 10 MB |
| Minimum size | Non-empty |
| Lost report photos | 0 or 1 |
| Found item photos | Exactly 1 |

Rejected uploads fail before storage is called. Unsupported media types map to
`415`, oversized payloads to `413`.

HEIC/HEIF is not a stored format. Phone uploads should be converted to JPEG
before upload, preferably in the frontend where the original file is available
and the user can get immediate feedback. Server-side HEIC conversion is avoided
because it would require image-processing/native decoding concerns that do not
belong in the storage layer.

## Retrieval

Services expose two retrieval styles after authorizing access to the owning
lost report or found item:

| Endpoint style | Behavior | Use |
| --- | --- | --- |
| Proxy download | The service reads the object through `PhotoStorage.retrieve(...)` and streams the bytes to the caller. | Fallback path and providers that cannot produce browser-reachable URLs. |
| Signed URL | The service calls `PhotoStorage.signedUrl(photoKey, ttl)` and returns `PhotoUrlResponse { "url": "..." }`. The browser downloads the bytes directly from MinIO or Azure Blob. | Preferred browser path for MinIO/Azure because application services do not proxy the photo bytes. |

The signed URL TTL is configured per service through:

```properties
PHOTO_STORAGE_SIGNED_URL_TTL=PT10M
```

`PT10M` is the default in both `lost-item-service` and `found-item-service`,
so generated URLs are valid for 10 minutes unless the environment overrides the
value. The TTL uses the ISO-8601 duration format accepted by Spring, for
example `PT5M`, `PT10M`, or `PT15M`.

The signed URL is intentionally short-lived. A caller must first pass the
service's JWT/venue authorization checks to receive it, but the returned URL can
then be used directly against the storage backend until it expires.

The local filesystem provider cannot create browser-reachable signed URLs and
therefore maps signed URL retrieval to `501 Not Implemented`. Callers should
use the proxy photo endpoint in that case.

## Error Handling

| Situation | Storage behavior | Service response |
| --- | --- | --- |
| Unknown key on retrieve or signed URL creation | `PhotoNotFoundException` | `404 Not Found` |
| Signed URL requested from local filesystem provider | `UnsupportedOperationException` | `501 Not Implemented` |
| Delete unknown key | No-op | `204 No Content` |
| Backend unavailable or store failure | `PhotoStorageException` | `502` or `503` |
| Upload validation failure | Storage is not called | `413` or `415` |

Delete is idempotent by contract, so retries can safely call delete again.
